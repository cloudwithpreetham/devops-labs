# What is Observability?

Observability is the ability to understand the internal state of a system by analyzing the data it produces.

Unlike traditional monitoring, which only alerts us when something goes wrong, observability helps us investigate and determine **why** something went wrong.

### Monitoring vs Observability

| Monitoring                   | Observability                                 |
| ---------------------------- | --------------------------------------------- |
| Detects known problems       | Helps investigate unknown problems            |
| Based on predefined alerts   | Enables exploration using collected telemetry |
| Answers **"What happened?"** | Answers **"Why did it happen?"**              |
| Threshold-based              | Correlation-based                             |

---

# The Three Pillars of Observability

## 1. Metrics

Metrics are numerical measurements collected over time.

Examples:

- CPU Usage
- Memory Usage
- Request Count
- Error Rate
- Disk Usage

Common Tools:

- Prometheus
- Datadog
- AWS CloudWatch

---

## 2. Logs

Logs are timestamped records generated by applications and infrastructure.

Examples:

- Application Logs
- Authentication Logs
- Error Logs
- Nginx Access Logs

Common Tools:

- Loki
- ELK Stack
- Fluentd

---

## 3. Traces

Traces show the complete journey of a request across multiple services.

They help identify:

- Slow services
- Failed requests
- Bottlenecks
- Network latency

Common Tools:

- OpenTelemetry
- Jaeger
- Zipkin

---

# Why DevOps Engineers Need All Three

| Pillar  | Helps Answer        |
| ------- | ------------------- |
| Metrics | What is broken?     |
| Logs    | Why is it broken?   |
| Traces  | Where is it broken? |

Example:

A user reports the application is slow.

Metrics show:

- High CPU usage

Logs show:

- Database connection timeout

Trace shows:

- Payment service took 12 seconds

Together they provide the complete picture.

# Prometheus Configuration

## prometheus.yml

```yaml
global:
  scrape_interval: 15s
  evaluation_interval: 15s

scrape_configs:
  - job_name: "prometheus"
    static_configs:
      - targets:
          - "localhost:9090"

  - job_name: "node-exporter"
    static_configs:
      - targets:
          - "node-exporter:9100"
```

---

# Docker Compose Configuration

```yaml
version: "3.8"

services:
  prometheus:
    image: prom/prometheus:latest
    container_name: prometheus
    restart: unless-stopped
    ports:
      - "9090:9090"
    volumes:
      - ./prometheus.yml:/etc/prometheus/prometheus.yml:ro
      - prometheus_data:/prometheus
    command:
      - "--config.file=/etc/prometheus/prometheus.yml"
      - "--storage.tsdb.retention.time=15d"
      - "--storage.tsdb.retention.size=1GB"

  node-exporter:
    image: prom/node-exporter:latest
    container_name: node-exporter
    restart: unless-stopped
    ports:
      - "9100:9100"
    command:
      - "--path.rootfs=/host"
    volumes:
      - "/:/host:ro,rslave"

volumes:
  prometheus_data:
```

# Prometheus Concepts Learned

## Scrape Target

A scrape target is an endpoint that exposes metrics.

Prometheus periodically pulls metrics from these targets.

---

## Time Series

A time series is uniquely identified by:

```
Metric Name + Labels
```

Example:

```
http_requests_total{method="GET",status="200"}
```

---

## Labels

Labels are key-value pairs that add dimensions to metrics.

Example:

```
method="GET"
status="200"
instance="localhost:9090"
```

---

# Metric Types

## Counter

A counter only increases.

Examples:

- Total HTTP Requests
- Total Errors
- Total Logins

Example Metric:

```
http_requests_total
```

---

## Gauge

A gauge increases and decreases.

Examples:

- CPU Usage
- Memory Usage
- Active Connections

Example Metric:

```
node_memory_MemAvailable_bytes
```

---

## Histogram

Measures value distributions using buckets.

Example:

- Request Duration
- Response Size

---

## Summary

Similar to Histogram but calculates quantiles on the client side.

---

# Counter vs Gauge

| Counter          | Gauge                    |
| ---------------- | ------------------------ |
| Only increases   | Can increase or decrease |
| Measures totals  | Measures current values  |
| Used with rate() | Never use rate()         |

Examples

Counter

```
http_requests_total
```

Gauge

```
node_memory_MemAvailable_bytes
```

# Prometheus Storage

Prometheus stores metrics inside its built-in Time Series Database (TSDB).

Default retention:

- 15 Days

Retention can be customized using:

```yaml
--storage.tsdb.retention.time=30d
--storage.tsdb.retention.size=1GB
```

---

# Why Persistent Volumes Are Important

Prometheus stores collected metrics on disk.

Without a Docker volume:

- All collected metrics are lost when the container is removed.

With a Docker volume:

- Metrics persist across container restarts.
- Historical data remains available.
- TSDB survives upgrades.

# Observation During the Lab

The provided `trainwithshubham/notes-app` container did not expose a `/metrics` endpoint.

Prometheus successfully connected to the application but received an HTTP **404 Not Found** response while scraping `/metrics`.

To continue the observability setup, I monitored Prometheus itself and Node Exporter, both of which expose native Prometheus metrics.

This demonstrated that Prometheus requires applications to expose metrics endpoints before they can be monitored.
